rust错误处理

Created2026-03-31|Updated2026-03-31|computer
|Post Views:

Rust 将错误分为两类：可恢复错误 和 不可恢复错误。

类型 处理方式 说明
可恢复错误 Result<T, E> 可以处理的错误，如文件不存在
不可恢复错误 panic! 程序无法继续执行的错误

不可恢复错误 (panic!)

Rust 的 panic 策略有两种：unwind（栈展开）和 abort（立即终止）。默认情况下，dev 和 release 配置都使用 unwind。如需使用 abort，可在 Cargo.toml 的 [profile] 中单独配置。两种策略对比如下表：

特性 unwind abort
行为 栈展开，清理资源 直接终止进程
可捕获 ✅ 可以（catch_unwind） ❌ 不可捕获
二进制大小 较大 较小（约减少 10-30%）
运行性能 略有开销 无额外开销
资源清理 执行析构函数 不执行，OS 回收
适用场景 开发调试、需要恢复 嵌入式、性能敏感

可恢复错误 (Result)

Result类型

基本使用

Result<T, E> 是 Rust 处理可恢复错误的核心类型。基础用法：

  • 使用 match 解构 Result
  • Ok(value) 分支获取成功值
  • Err(error) 分支处理错误
  • 错误分类：对于标准库的错误（如 std::io::Error），可通过 .kind() 获取错误类型；对于自定义错误，直接匹配错误内容

注意上面的file_result其实是一个Result<File>类型，它其实是一种类型别名，Rust 标准库中许多模块都定义了 Result 的类型别名，将错误类型固定为模块特定的错误。例如：

  • std::io::Result<T> = Result<T, std::io::Error>
  • std::fs::Result<T> = Result<T, std::io::Error>

这样使用时只需写 io::Result<T>，代码更简洁，同时保持了错误类型的明确性。

常用方法

错误传播

错误传播（? 操作符）

? 操作符用于简化错误传播，避免嵌套 match。当函数返回 Result 或 Option 时：

  • 如果 ? 作用于 Ok(value)，则返回 value 继续执行
  • 如果作用于 Err(e)，则自动 return Err(e.into())，将错误向上传播

核心特点：

  1. 自动转换：? 会调用 From::from 将错误类型转换为函数返回的错误类型
  2. 使用限制：只能在返回 Result 或 Option 的函数中使用
  3. 链式调用：支持连续使用，代码更简洁

自定义错误类型

简单错误类型

Rust 中创建自定义错误需要实现 std::fmt::Display 和 std::error::Error 两个 trait。Display 用于格式化输出，Error 标记该类型可作为错误使用。

核心要点：

  • 结构体定义：包含错误码和错误信息，便于分类处理
  • new 方法：构造函数，简化错误创建，是 Rust 的惯用模式
  • kind 方法：扩展方法，根据错误码返回错误类型，方便调用方分类处理
  • Display 实现：必须实现，定义错误的展示格式（如 println! 时显示的内容）
  • Error trait：可选但推荐实现，标记为错误类型，支持 ? 传播和 Box<dyn Error>
组成部分 作用 是否必须
struct 定义错误数据结构 ✅ 必须
impl 构造函数 简化错误创建 ❌ 推荐
impl 扩展方法 提供错误分类等辅助功能 ❌ 按需
Display 定义错误如何显示 ✅ 必须
Error trait 标记为错误类型，支持 ? 传播 ✅ 推荐

枚举错误类型

当函数可能产生多种不同类型的错误时，使用枚举来统一错误类型是最佳实践。这种方式可以清晰地区分错误来源，并通过 From 转换实现 ? 操作符的自动转换。

核心要点：

  • 枚举定义：为每种错误类型定义一个变体，可以是标准错误（如 io::Error）或自定义错误（如 Custom(String)）
  • Display 实现：为每个变体定义合适的输出格式，方便错误展示
  • Error trait：标记为错误类型，使其可用于 ? 传播
  • From 转换：为每个标准错误类型实现 From trait，使得 ? 操作符可以将具体错误自动转换为枚举类型

最大优势：通过实现 From，代码中可以直接使用 ?，无需手动 map_err，保持代码简洁。

Box<dyn Error>

Box<dyn std::error::Error> 是 Rust 标准库提供的抽象错误类型，可以容纳任何实现了 Error trait 的错误。

核心特点：

  • 万能容器：可以存放任何具体错误类型（io::Error、ParseIntError、自定义错误等）
  • 自动转换：通过 ? 操作符，具体错误会自动转换为 Box<dyn Error>
  • 动态分发：使用 trait 对象，有轻微性能开销
  • 有限信息：丢失具体类型，需要 downcast 才能恢复

使用场景：

  • main 函数快速返回错误
  • 测试代码
  • 快速原型开发
  • 不需要精确处理错误类型的场景

第三方错误处理库

anyhow

anyhow 是 Rust 生态中用于应用层错误处理的常用库，提供便捷的 anyhow::Result<T> 类型别名，可以容纳任何实现了 Error trait 的错误类型。

核心特点：

  • anyhow::Result<T>：等同于 Result<T, anyhow::Error>，是 Box<dyn Error> 的增强版
  • anyhow! 宏：快速创建带上下文的错误信息，支持格式化字符串
  • 自动转换：任何实现了 Error 的错误都可以用 ? 自动转换为 anyhow::Error
  • 丰富上下文：支持添加错误链，方便调试

适用场景：

  • 二进制应用（CLI 工具、Web 服务）
  • 快速原型开发
  • 不需要精确区分错误类型的场景

代码汇总

thiserror

thiserror 是用于库开发的错误处理库，通过过程宏自动生成 Display 和 From 等实现，极大简化自定义错误类型的编写。

核心特点：

  • #[derive(Error)]：自动实现 std::error::Error trait
  • #[error("...")]：定义错误的 Display 输出格式
  • #[from]：自动生成 From 转换，让 ? 可以自动包装错误
  • 支持多种变体：元组结构体、具名字段、无字段枚举等

适用场景：

  • 库开发（需要明确错误类型供调用方处理）
  • 需要精确区分错误类型的大型项目
  • 与 anyhow 配合：库用 thiserror，应用用 anyhow

Option

Option 用于表示值可能存在也可能不存在的情况，是 Rust 处理可选值的核心类型。与 Result 不同，Option 不提供错误信息，只表示"有"或"无"。
