线程

Rust 的线程模型是 1:1 的，即一个 Rust 线程对应一个操作系统线程。理解线程的生命周期，是写好并发程序的第一步。

基础创建

关键理解：join() 是一个阻塞操作，它会等待线程执行完毕。如果忘记调用 join()，主线程结束时会自动 detach 子线程，子线程可能无法完成。

线程配置

通过 Builder 你可以对线程的初始状态进行更多的控制，比如设置线程的名称、栈大大小等等

什么时候需要配置栈大小？

  • 递归深度很大（如深度优先搜索）
  • 栈上分配大量数据
  • 嵌入式环境内存受限

唤醒机制

  • thread::park()：阻塞当前线程，使其进入等待状态
  • thread::park_timeout()：带超时的阻塞，被唤醒或超时都会解除阻塞
  • thread::unpark()：唤醒被阻塞的线程

所有权转移

move 关键字将外部变量的所有权转移到闭包内部，这是 Rust 线程安全的基石。

为什么需要 move？ 因为新线程的生命周期可能超过当前作用域，编译器必须保证线程访问的数据在它运行期间始终有效。转移所有权是最直接的解决方案。

作用域线程

Rust 1.63 引入的 scope API 解决了 move 过于严格的问题：

原理：scope 会阻塞当前线程，直到作用域内所有子线程执行完毕。正是这个特性，让编译器可以放心地允许借用数据

作用域线程 vs 普通线程

维度 普通线程 作用域线程
生命周期 可能超过创建者 被限制在作用域内
数据访问 只能 move 可以借用
自动管理 需要手动 join 自动等待
适用场景 长生命周期任务 短期并行计算

控制线程

标准库提供的线程控制能力非常有限：

  • 没有内置方法从外部安全地停止一个线程
  • 没有方法检查线程是否还在运行
  • 没有方法检测线程是否 panic 了

thread-control 填补了这个空白，提供了一个轻量级的线程控制机制，特别适合需要长时间运行循环并能够优雅停止的场景

make_pair 是 thread-control 库提供的函数这个函数的作用是创建一个线程控制对 (pair)，返回两个对象：

  • Flag：在线程内部使用，用于检查线程是否应该继续运行
  • Control：在其他线程中使用，用于控制/停止目标线程

Rayon

Rayon 是 Rust 生态中最流行的数据并行计算库，核心目标是：让并行编程变得像普通迭代器一样简单。

它的设计哲学是：你只需要告诉 Rayon “这些数据要并行处理”，剩下的任务拆分、线程调度、结果合并全部由库自动完成。

特点 说明
极简 API 把 .iter() 改成 .par_iter() 即可并行
工作窃取调度 自动负载均衡，避免线程空闲或过载
内置线程池 默认使用 CPU 核心数个线程，无需手动管理
安全并行 利用 Rust 所有权系统，编译期避免数据竞争

线程间通信

核心哲学：“不要通过共享内存来通信，而要通过通信来共享内存” —— Go 语言的并发哲学，Rust 同样推崇。

通道的本质：通道是多线程间传递数据的安全机制，发送者将数据送进通道，接收者从通道取出数据。数据的所有权在传递过程中发生了转移，因此无需担心数据竞争。

mpsc

mpsc = Multiple Producer, Single Consumer（多生产者，单消费者）。这是 Rust 标准库提供的基础通道类型。

接收端的三种接收方式

方法 阻塞行为 无消息时 通道关闭时
recv() ✅ 阻塞等待 一直等直到有消息 返回 Err(Disconnected)
try_recv() ❌ 不阻塞 立即返回 Err(Empty) 立即返回 Err(Disconnected)
recv_timeout() ⏱️ 超时阻塞 等待指定时间，超时返回 Err(Timeout) 返回 Err(Disconnected)

注意：这三个方法一次只接收一条消息，需要循环才能接收多条。

⚠️ 常见错误：忘记 drop(tx) 原始发送者。此时还有一个发送者存活（尽管不发送数据），接收者会永远等待下去。

同步阻塞

同步阻塞指的是：当通道容量已满时，发送者会被阻塞，直到有消费者取走数据，腾出空间。

  • 容量选择指南：

    通道类型 创建方式 行为 适用场景
    容量 0 sync_channel(0) 完全同步，发送和接收必须同时发生（会合点） 精确的线程同步
    容量 N sync_channel(N) 缓冲 N 条消息，满了才阻塞 生产速度波动、限流
    无界通道 channel() 永不阻塞，无限增长 速度差异不大、可控场景

    ⚠️ 无界通道风险：如果消费者跟不上，消息会无限堆积，可能导致内存耗尽。

多消费者

标准库的 mpsc 只支持单消费者，crossbeam 提供了更灵活的通道：

crossbeam 通道特性：

  • 支持多生产者多消费者（MPMC）
  • 性能与标准库相当，但功能更丰富
  • 提供 select! 宏，可以同时监听多个通道

共享状态的并发

核心哲学：通道是"通过通信来共享内存"，而互斥锁是"通过共享内存来通信"。两种方式各有优劣，根据场景选择。

何时用锁？ 当多个线程需要同时读写同一份数据时（如共享缓存、全局计数器），锁是最直接的选择。

互斥锁（Mutex）

Mutex 是 Mutual Exclusion（互斥）的缩写，确保同一时刻只有一个线程能访问被保护的数据。其他线程想要访问就必须阻塞等待，直到锁被释放。

关键点说明：

组件 作用
Mutex<T> 保护数据 T，提供 lock() 方法
lock() 返回 MutexGuard，获取失败时返回 Err（中毒）
MutexGuard 智能指针，解引用后访问数据，drop 时自动释放锁
Arc<T> 原子引用计数，让多个线程共享 Mutex 的所有权

锁的自动管理（RAII）：

  • MutexGuard 离开作用域时自动释放锁，无需手动调用 unlock
  • 即使线程 panic，锁也会被释放（但会进入"中毒"状态）

锁持有时间原则：

锁持有时间越长，并发性能越差。应该只在对共享数据访问时持有锁：

锁中毒（Poisoning）：

当线程在持有锁期间 panic，锁会进入"中毒"状态。后续获取锁的 lock().unwrap() 会 panic，因为数据可能处于不一致状态。

读写锁（RwLock）

RwLock 提供了更细粒度的并发控制：

  • 多个读锁可以同时存在（共享访问），读操作不互斥
  • 写锁是独占的（排他访问），写操作时不允许任何读写

读锁与写锁的互斥关系：

操作 读锁 写锁
读锁 ✅ 可同时 ❌ 阻塞等待
写锁 ❌ 阻塞等待 ❌ 阻塞等待

性能选择指南：

场景 推荐锁 原因
读操作占比 > 90% RwLock 读并发高，写少，充分发挥共享读优势
读写均衡或写多 Mutex RwLock 有额外开销，写操作时并不比 Mutex 快

RwLock 的开销来源：

  • 需要维护读锁计数（原子操作）
  • 写锁需要等待所有读锁释放
  • 内部实现比 Mutex 更复杂（需要区分读写状态）

💡 经验法则：不确定时先用 Mutex。只有当明确读多写少且成为性能瓶颈时，才考虑换成 RwLock。

条件变量（Condvar）

条件变量让线程等待某个条件成立时被唤醒，避免忙等待（busy loop）浪费 CPU。

典型场景：生产者-消费者模式中，消费者需要等待队列非空；或者线程需要等待某个标志位被设置。

为什么需要 while 循环？

可能存在虚假唤醒（spurious wakeup），线程可能在没有收到通知的情况下被唤醒。用 while 重新检查条件，确保条件真正满足后才继续执行。

Condvar 的核心方法：

方法 作用
wait(guard) 释放锁并阻塞，被唤醒后重新获取锁
notify_one() 唤醒一个等待的线程
notify_all() 唤醒所有等待的线程

💡 注意：Condvar 总是需要与 Mutex 配合使用，因为需要保护条件变量的共享状态。

屏障（Barrier）

屏障让多个线程在同一时刻汇合，所有线程到达屏障后才一起继续执行。适用于多阶段并行计算，每个阶段需要所有线程完成后才能进入下一阶段。

无锁并发

原子操作是CPU 级别的不可分割操作，是无锁编程的基础。相比互斥锁，原子操作：

  • 无阻塞：不会导致线程休眠和上下文切换
  • 更高效：适用于简单计数、标志位等场景
  • 更复杂：需要理解内存模型和指令重排

内存顺序

内存顺序是 Rust 原子操作中最复杂的部分，它告诉 CPU 和编译器如何重排指令

顺序 保证 性能 适用场景
Relaxed 仅保证原子性 最快 简单计数
Acquire-Release 建立同步关系 中等 生产者-消费者
SeqCst 全局一致顺序 最慢 复杂同步

Relaxed

Relaxed 最宽松得顺序，只保证单个变量的原子性，不保证操作的顺序。

适用场景：

  • 简单的计数器递增（不依赖其他数据）
  • 统计指标收集
  • 不涉及同步关系的标志位

Release-Acquire

Release 和 Acquire 配对使用，建立"先于"（happens-before）关系。

核心概念

  • Release（释放）：确保之前的所有操作不会重排到 store 之后
  • Acquire（获取）：确保之后的所有操作不会重排到 load 之前

SeqCst

SeqCst（Sequentially Consistent）在所有线程中保持一致的全局顺序。

SeqCst 在所有线程中保持一致的全局顺序，是最直观但性能最差的内存顺序。
