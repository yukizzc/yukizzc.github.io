线程的生命周期

理解线程的诞生与消亡

Rust 的线程模型是 1:1 的，即一个 Rust 线程对应一个操作系统线程。理解线程的生命周期，是写好并发程序的第一步。

基础创建

关键理解：join() 是一个阻塞操作，它会等待线程执行完毕。如果忘记调用 join()，主线程结束时会自动 detach 子线程，子线程可能无法完成。

所有权转移

move 关键字将外部变量的所有权转移到闭包内部，这是 Rust 线程安全的基石。

为什么需要 move？ 因为新线程的生命周期可能超过当前作用域，编译器必须保证线程访问的数据在它运行期间始终有效。转移所有权是最直接的解决方案。

作用域线程

Rust 1.63 引入的 scope API 解决了 move 过于严格的问题：

原理：scope 会阻塞当前线程，直到作用域内所有子线程执行完毕。正是这个特性，让编译器可以放心地允许借用数据

作用域线程 vs 普通线程

维度 普通线程 作用域线程
生命周期 可能超过创建者 被限制在作用域内
数据访问 只能 move 可以借用
自动管理 需要手动 join 自动等待
适用场景 长生命周期任务 短期并行计算

同步阻塞

同步阻塞指的是：当通道容量已满时，发送者会被阻塞，直到有消费者取走数据，腾出空间。

容量选择：

  • 容量 0：生产者消费者完全同步（会合点）
  • 容量 N：缓冲 N 条消息，满了才阻塞
  • 无界通道：永不阻塞，但有内存风险，就是前面的 mpsc::channel()

共享状态的并发

互斥锁

互斥锁确保同一时刻只有一个线程能访问数据。

锁的自动管理：

  • lock() 返回 MutexGuard：这是一个智能指针，解引用后可访问内部数据
  • 自动释放：MutexGuard 离开作用域时自动释放锁（RAII 原则）
  • Panic 处理：线程 panic 时锁会被释放，但变成 poisoned（中毒）状态

锁持有时间越长，并发性能越差。应该只在对共享数据访问时持有锁

读写锁

RwLock 允许：

  • 多个读锁同时存在（共享访问）
  • 写锁独占（排他访问）

性能考虑

场景 推荐锁 原因
读操作占比 > 90% RwLock 读并发高，写少
读写均衡或写多 Mutex RwLock 有额外开销，写操作时并不比 Mutex 快

RwLock 的开销来源：

  • 需要维护读锁计数
  • 写锁需要等待所有读锁释放
  • 内部实现比 Mutex 复杂

条件变量

条件变量让线程等待某个条件成立时被唤醒，避免忙等待（busy loop）浪费 CPU。

为什么需要 while 循环？ 因为可能存在"虚假唤醒"，线程可能在没有收到通知的情况下被唤醒。

屏障

屏障让多个线程在同一时刻汇合，所有线程到达屏障后才继续执行。

无锁并发

原子操作是CPU 级别的不可分割操作，是无锁编程的基础。相比互斥锁，原子操作：

  • 无阻塞：不会导致线程休眠和上下文切换
  • 更高效：适用于简单计数、标志位等场景
  • 更复杂：需要理解内存模型和指令重排

内存顺序

内存顺序是 Rust 原子操作中最复杂的部分，它告诉 CPU 和编译器如何重排指令

顺序 保证 性能 适用场景
Relaxed 仅保证原子性 最快 简单计数
Acquire-Release 建立同步关系 中等 生产者-消费者
SeqCst 全局一致顺序 最慢 复杂同步

Relaxed

Relaxed 最宽松得顺序，只保证单个变量的原子性，不保证操作的顺序。

适用场景：

  • 简单的计数器递增（不依赖其他数据）
  • 统计指标收集
  • 不涉及同步关系的标志位

Release-Acquire

Release 和 Acquire 配对使用，建立"先于"（happens-before）关系。

核心概念

  • Release（释放）：确保之前的所有操作不会重排到 store 之后
  • Acquire（获取）：确保之后的所有操作不会重排到 load 之前

SeqCst

SeqCst（Sequentially Consistent）在所有线程中保持一致的全局顺序。

SeqCst 在所有线程中保持一致的全局顺序，是最直观但性能最差的内存顺序。

Rayon

Rayon 是 Rust 生态中最流行的数据并行（data-parallelism）库，它的口号是：“把顺序计算变成并行计算，简单到令人发指” 。

  1. 极简 API：通常只需要把 .iter() 改成 .par_iter()
  2. 安全保证：编译通过 = 没有数据竞争（data race）
  3. 自动调度：自动决定如何分割任务、如何分配线程
  4. 高性能：基于 work-stealing 算法，动态负载均衡
