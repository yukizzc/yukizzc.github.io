异步编程基础

异步编程是一种不阻塞线程的并发模型。当程序等待 I/O 操作（如网络请求、文件读写）时，不会让当前线程空等，而是去执行其他任务，等 I/O 完成后回来继续执行。

维度 同步 多线程 异步
等待 I/O 时 线程阻塞 该线程阻塞，其他线程继续 线程切换去执行其他任务
线程数量 1 个 N 个（通常每个任务一个） 少量（通常等于 CPU 核心数）
上下文切换 无 频繁（操作系统级） 少（用户态任务切换）
内存开销 低 高（每个线程有独立栈，通常 MB 级） 低（每个任务只需 KB 级）
适用场景 简单任务 CPU 密集型 I/O 密集型
并行能力 无 真正并行（多核） 需要配合多线程运行时

核心理解：异步的价值在于高并发而非高性能计算。一个线程通过任务调度可以管理成千上万个网络连接，这在同步模型中需要成千上万个线程才能做到。

Rust 异步的特点

特点 说明
零成本抽象 异步代码编译后与手写状态机一样高效，没有运行时开销
无内置运行时 标准库只提供 Future trait，具体运行时（tokio、async-std）由开发者选择
基于 async/await 语法类似同步代码，易于编写和理解
与所有权系统集成 编译时检查借用和生命周期，避免数据竞争

Future 核心概念

Future trait 的本质

Future 是一个状态机，代表一个可能尚未完成的计算：

poll 机制

Future 不是自动执行的，而是需要被不断 poll（轮询）：

  1. 执行器调用 poll() 尝试推进 Future
  2. 如果返回 Pending，执行器会等待唤醒通知
  3. 当任务可以继续时（如 I/O 完成），waker 通知执行器再次 poll
  4. 直到返回 Ready(T)

异步函数

async fn 会被编译器转换成一个返回 impl Future 的状态机

.await

.await 会不断 poll 一个 Future，直到它完成

关键点：.await 会出让当前任务的执行权，而不是阻塞线程。

运行时

Rust 标准库只提供了异步编程的基础设施：

组件 作用 说明
Future trait 定义异步计算 提供 poll() 方法，由执行器调用
Waker 机制 任务唤醒 Future 通知执行器"我可以继续执行了"
Pin 保证内存稳定 异步块中可能包含自引用结构

关键点：标准库没有提供执行器（Executor），也没有 I/O 事件循环。需要额外的运行时来驱动。

一个完整的异步运行时需要提供：

  1. Executor（执行器）：管理任务队列，调度 Future 执行
  2. Reactor（反应器）：监听 I/O 事件，在资源就绪时唤醒任务
  3. Timer（定时器）：处理 sleep、timeout 等时间相关操作

  • 90% 的场景：直接用 tokio，它是 Rust 异步的事实标准
  • 学习异步概念：async-std 更直观
  • 极简/嵌入式：smol 更轻量
  • 极致性能：glommio 或 monoio（需了解 io_uring）

异步任务

重要前提：async 函数本身不会并发执行。它们只是返回一个 Future，代表一个可以被执行的计算。如果不 spawn 或者 .await，这个 Future 什么都不会做

关键区别：

写法 行为 并发性
future.await 当前任务等待 Future 完成 ❌ 顺序执行，阻塞当前任务
tokio::spawn(future) 创建独立任务，立即返回 JoinHandle ✅ 并发执行，不阻塞当前任务

spawn

spawn 会在运行时中创建一个新的独立任务，这个任务会并发执行，与当前任务互不阻塞。

适用场景：当你需要将任务放到后台独立运行，且可能需要等待其结果时

spawn 的关键规则：

规则 说明
任务立即开始 spawn 后任务进入就绪队列，但不一定立即完成
返回 JoinHandle 调用 .await 等待任务完成并获取返回值
Panic 传播 任务 panic 时，JoinHandle::await 返回 Err
并发执行 多个 spawn 的任务并发执行，而非顺序执行

join!

join! 会同时执行多个 Future，并等待所有都完成后才返回。它不创建新的任务，而是在当前任务中并发轮询多个 Future。

select!

select! 会同时执行多个 Future，但只等待第一个完成的那个，其余 Future 会被丢弃或继续运行（取决于是否使用 else）。

适用场景：当你有多个备选操作，只要其中一个完成就可以继续时（如超时控制、多源数据竞争）。

对比总结

特性 spawn join! select!
执行位置 独立的后台任务 当前任务中并发 当前任务中并发
等待策略 需要手动 .await handle 等待所有完成 等待第一个完成
返回值 JoinHandle<T> 元组 (T1, T2, ...) 对应分支的表达式
能否取消 ✅ drop handle 即可 ❌ 无法单独取消 ✅ 未完成的分支被 drop
典型场景 后台任务、长时间运行 并行请求多个 API 超时控制、竞速选择
开销 较高（任务分配） 较低 较低

选择指南

场景 推荐方式
需要任务在后台独立运行，不关心何时完成 spawn
需要并发请求多个服务，等待所有结果 join!
需要设置超时，或从多个数据源中竞争最快响应 select!
需要动态创建大量任务 spawn
只是简单并发执行几个已知的 Future join!（更轻量）

异步原语

异步环境下的同步原语与同步版本的关键区别在于：它们支持 .await，在等待锁或消息时不会阻塞整个线程，而是让出执行权给其他任务，从而实现高并发。

通道

通道用于任务间消息传递，遵循"通过通信来共享内存"的理念。

mpsc

最常用的通道类型，适合多个任务向同一个接收端发送消息的场景（如工作队列、日志收集）

方法 说明
send() 异步发送，通道满时等待（背压控制）
recv() 异步接收，无消息时等待
try_send() 立即尝试发送，不阻塞
try_recv() 立即尝试接收，不阻塞

典型场景：任务队列、日志管道、流式数据处理。

oneshot

一对一的单次消息传递，发送端和接收端各一个，只能发一次消息。语义清晰，适合需要明确回复的场景

适用场景：请求-响应模式（如 HTTP 请求的响应通道）、任务完成通知、错误处理中的一次性反馈

broadcast

广播通道，每个消息会发送给所有活跃的订阅者。类似消息队列中的 pub/sub 模式。

特点：

  • 每个消费者收到相同的消息（而非负载均衡）
  • 新订阅者不会收到历史消息
  • 通道满时，最旧的消息会被丢弃（可配置）

典型场景：实时行情推送、聊天室消息、配置变更通知。

watch

状态广播通道，只保留最新的值，新订阅者立即获得当前状态。与 broadcast 的区别是：它关注的是"当前状态"而非"每条消息"。

特点：

  • 只保留最新值，历史消息不保留
  • 新订阅者立即获得当前值
  • 支持多生产者和多消费者

典型场景：配置热更新、系统状态监控、进度条展示

互斥锁

异步版本的互斥锁，关键区别是 lock() 是异步的，在等待锁时可以让出执行权给其他任务，而不是阻塞整个线程。

多任务共享

和 std::Mutex 一样，tokio::Mutex 也不能被多个任务直接共享，必须用 Arc 包装

正确做法

tokio::Mutex vs std::Mutex

如果在std::mutex获取锁的情况下调用.await则会发生

  1. 获取锁：guard 拿到锁，保护数据 42
  2. 遇到 .await：任务让出执行权，去执行其他任务
  3. 锁没有被释放：guard 还在作用域内，锁一直被持有
  4. 其他任务想获取锁：如果其他任务也想获取这个锁，会被阻塞
  5. 原任务无法释放锁：原任务正在等待 some_async() 完成，处于"等待 I/O"状态，无法继续执行来释放锁
  6. 结果：其他任务可能被长时间阻塞，形成性能瓶颈甚至死锁

执行过程详解

  1. 获取锁：guard 拿到锁
  2. 遇到 .await：任务让出执行权
  3. 锁被释放：tokio::Mutex 的 guard 检测到要进入 .await，主动释放锁
  4. 其他任务可以获取锁：其他任务可以正常获取锁并执行
  5. 原任务恢复：some_async() 完成后，原任务被唤醒，重新获取锁
  6. 继续执行：获取锁后继续执行后续代码
