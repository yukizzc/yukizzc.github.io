异步编程基础

异步编程允许程序在等待 I/O 操作（如网络请求、文件读写）时，不阻塞当前线程，而是去执行其他任务，等 I/O 完成后回来继续执行。

同步 vs 异步 vs 多线程

维度 同步 多线程 异步
等待 I/O 时 线程阻塞 线程阻塞 线程切换任务
线程数量 1 个 N 个（每个任务一个） 少量（通常等于 CPU 核心数）
上下文切换 无 频繁（线程间） 少（任务间）
内存开销 低 高（每个线程有独立栈） 低（任务共享线程）
适用场景 简单任务 CPU 密集型 I/O 密集型

Rust 异步的特点

  • 零成本抽象：异步代码编译后与手写状态机一样高效
  • 无运行时强制：核心功能由标准库提供，具体运行时可选（tokio、async-std 等）
  • 基于 Future 和 .await：语法类似同步代码，易于编写和理解

Future 核心概念

Future trait 的本质

Future 是一个状态机，代表一个可能尚未完成的计算：

poll 机制

Future 不是自动执行的，而是需要被不断 poll（轮询）：

  1. 执行器调用 poll() 尝试推进 Future
  2. 如果返回 Pending，执行器会等待唤醒通知
  3. 当任务可以继续时（如 I/O 完成），waker 通知执行器再次 poll
  4. 直到返回 Ready(T)

异步函数

async fn 会被编译器转换成一个返回 impl Future 的状态机

.await

.await 会不断 poll 一个 Future，直到它完成

关键点：.await 会出让当前任务的执行权，而不是阻塞线程。

运行时

Rust 标准库只提供了异步编程的基础设施：

组件 作用 说明
Future trait 定义异步计算 提供 poll() 方法，由执行器调用
Waker 机制 任务唤醒 Future 通知执行器"我可以继续执行了"
Pin 保证内存稳定 异步块中可能包含自引用结构

关键点：标准库没有提供执行器（Executor），也没有 I/O 事件循环。需要额外的运行时来驱动。

一个完整的异步运行时需要提供：

  1. Executor（执行器）：管理任务队列，调度 Future 执行
  2. Reactor（反应器）：监听 I/O 事件，在资源就绪时唤醒任务
  3. Timer（定时器）：处理 sleep、timeout 等时间相关操作

Rust 生态中的异步运行时

运行时 特点 适用场景 维护状态
tokio 功能最全、生态最好、文档完善 Web 服务、数据库、网络应用 ✅ 活跃，事实标准
async-std API 与标准库一致，学习曲线低 快速原型、教育项目 ⚠️ 维护减缓
smol 极简设计、编译快、启动快 嵌入式、CLI 工具 ✅ 稳定维护
glommio 线程专用、共享队列、高吞吐 高性能存储系统 ✅ 活跃，小众
monoio 基于 io_uring，Linux 6+ 优化 极致 IO 性能场景 ✅ 活跃，阿里出品

选择建议：

  • 90% 的场景：直接用 tokio，它是 Rust 异步的事实标准
  • 学习异步概念：async-std 更直观
  • 极简/嵌入式：smol 更轻量
  • 极致性能：glommio 或 monoio（需了解 io_uring）

spawn 的规则：

  • 任务会立即开始执行（但不一定立即完成）
  • spawn 返回 JoinHandle，调用 .await 等待任务完成
  • 任务如果 panic，JoinHandle::await 会返回 Err

异步原语

异步环境下的同步原语与同步版本的关键区别在于：它们支持 .await，在等待锁或消息时不会阻塞整个线程，而是让出执行权给其他任务，从而实现高并发。

通道

通道用于任务间消息传递，遵循"通过通信来共享内存"的理念。

mpsc

最常用的通道类型，适合多个任务向同一个接收端发送消息的场景（如工作队列、日志收集）

方法 说明
send() 异步发送，通道满时等待（背压控制）
recv() 异步接收，无消息时等待
try_send() 立即尝试发送，不阻塞
try_recv() 立即尝试接收，不阻塞

典型场景：任务队列、日志管道、流式数据处理。

oneshot

一对一的单次消息传递，发送端和接收端各一个，只能发一次消息。语义清晰，适合需要明确回复的场景

适用场景：请求-响应模式（如 HTTP 请求的响应通道）、任务完成通知、错误处理中的一次性反馈

broadcast

广播通道，每个消息会发送给所有活跃的订阅者。类似消息队列中的 pub/sub 模式。

特点：

  • 每个消费者收到相同的消息（而非负载均衡）
  • 新订阅者不会收到历史消息
  • 通道满时，最旧的消息会被丢弃（可配置）

典型场景：实时行情推送、聊天室消息、配置变更通知。

watch

状态广播通道，只保留最新的值，新订阅者立即获得当前状态。与 broadcast 的区别是：它关注的是"当前状态"而非"每条消息"。

特点：

  • 只保留最新值，历史消息不保留
  • 新订阅者立即获得当前值
  • 支持多生产者和多消费者

典型场景：配置热更新、系统状态监控、进度条展示

互斥锁

异步版本的互斥锁，关键区别是 lock() 是异步的，可以在等待锁时让出执行权

tokio::Mutex vs std::Mutex：

场景 推荐 原因
锁持有时间短，不包含 .await std::Mutex 开销更小，性能更好
锁持有期间包含 .await tokio::Mutex 避免阻塞整个线程
跨多个 .await 点持有锁 tokio::Mutex 线程安全，可被唤醒

原则：除非确实需要在 .await 点持有锁，否则优先使用 std::sync::Mutex，它的开销更小。

异步任务管理

核心理解：异步不是并行，而是协作式多任务。一个线程可以管理成千上万个任务，当一个任务在等待 I/O 时，它主动让出执行权（通过 .await），运行时调度另一个任务继续执行。这种"让出"机制，使得单线程也能高效处理大量并发连接。

概念 本质 执行方式
同步 顺序执行 一个任务等另一个任务完成
异步 非阻塞等待 等待时让出执行权，但不一定并行
并行 同时执行 多个任务真正同时运行（需要多核）

异步的真正价值：在单线程上也能实现高并发，因为它通过任务调度和协作式让出，让一个线程"同时"处理成千上万个任务。
