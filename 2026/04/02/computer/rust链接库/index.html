概念

特性 静态链接库 动态链接库
链接时机 编译时 运行时
代码整合 库代码被直接复制到最终的可执行文件中。 库代码存在于独立的文件中，可执行文件只记录依赖关系。
文件后缀 Linux/Unix: .a (Archive) Windows: .lib Linux/Unix: .so (Shared Object) Windows: .dll (Dynamic Link Library)
可执行文件大小 较大，因为包含了所有依赖的库代码。 较小，因为不包含库代码本身。
内存占用 可能更高，因为每个使用该库的程序都在内存中有自己的副本。 可能更低，因为多个程序可以共享内存中的同一份库代码。
部署便利性 简单。可执行文件是独立的，不需要额外文件。 复杂。必须确保目标系统上有正确版本的库文件。
版本更新 困难。需要重新编译整个程序才能更新库。 容易。替换 DLL/.so 文件即可更新库，但需注意版本兼容性。
兼容性问题 无（因为库已绑定）。 可能存在“DLL Hell”，即版本冲突或缺失。

Rust 静态库（rlib）

Rust 生态的默认方式：rlib 是 Rust 编译器生成的中间库格式，专门用于 Rust 项目之间的依赖。但 Rust 的最佳实践是通过 Cargo 的依赖管理（Cargo.toml）直接引用源代码，而不是直接分发和手动链接 rlib 文件。

为什么不直接分发 rlib 文件？

问题 说明
编译器版本依赖 rlib 依赖特定 Rust 版本，不同版本编译器可能不兼容
平台相关 不同目标平台需要不同的 rlib 文件
生态习惯 Rust 通过 crates.io 分发源码，Cargo 统一编译管理

编译过程

当一个 Rust 二进制项目通过 Cargo 编译时，其构建过程如下：

  1. 依赖解析：Cargo 读取 Cargo.toml 中声明的依赖项，定位到库的源代码目录。
  2. 编译库依赖：Cargo 编译所依赖的库项目，生成 .rlib 文件（临时中间产物）。
  3. 静态链接：Cargo 将 .rlib 文件中的代码静态链接到最终的可执行文件中。
  4. 完成：一旦链接完成，生成可执行文件后，.rlib 文件的任务也就结束了，它不参与程序的运行。

动态库（cdylib）

若需创建动态链接库，推荐使用 cdylib 而非 dylib，并通过 extern "C" 定义接口，使用 C 兼容类型（如 i32、f64 或 #[repr(C)] 结构体）

为什么不推荐直接使用 Rust 的 dylib？

问题 说明
ABI 不稳定 Rust 的 ABI 没有稳定的保证，不同编译器版本可能生成不兼容的动态库
复杂类型问题 String、Vec、Option 等类型无法直接跨库传递，内存布局可能变化
运行时加载复杂 Rust 标准库不提供动态加载机制，需要依赖第三方库（如 libloading）
跨语言兼容性差 其他语言无法直接调用 Rust 的 dylib

cdylib 的优势

  • C 兼容性：生成标准的 C 风格动态链接库，可被其他语言直接调用
  • ABI 稳定：通过 extern "C" 和 #[unsafe(no_mangle)] 确保接口稳定
  • 跨平台支持：与操作系统原生动态库格式兼容

📌 核心定位：cdylib 专用于 Rust ↔ 其他语言的跨语言调用
📌 Rust ↔ Rust 代码复用，请使用静态链接（Workspace + 源码依赖），不要用动态库
