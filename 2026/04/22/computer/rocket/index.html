框架概述

Rocket 是 Rust 生态中最成熟、最易用的 Web 框架之一。它的设计理念是：让编写安全、高效的 Web 应用变得简单愉快。

与传统的 Web 框架不同，Rocket 充分利用了 Rust 的所有权系统和类型系统，能在编译时就发现很多潜在的错误。比如，你永远不会在路由函数中收到错误类型的参数，因为编译器会强制检查。

特性 Rocket的实现 带来的好处
声明式路由 使用属性宏 #[get("/")] 路由与处理函数绑定清晰，易于维护
类型安全的参数提取 参数类型自动转换 编译时保证类型正确，避免运行时错误
请求守卫 实现了 FromRequest 的类型 自动验证和提取请求数据，减少样板代码
响应器 实现了 Responder 的类型 灵活控制响应格式和状态码
中间件系统 Fairing（整流罩） 可在请求生命周期多个阶段插入逻辑

路由系统

路由是 Web 框架的核心——它决定了"哪个 URL 由哪个函数处理"。

HTTP方法绑定

Rocket 为每个 HTTP 方法都提供了对应的宏：

设计理念：RESTful API 通常遵循"用 HTTP 方法表达操作意图"的原则：

  • GET：获取资源（不应该修改数据）
  • POST：创建新资源
  • PUT：完整替换资源
  • PATCH：部分更新资源
  • DELETE：删除资源

动态路径参数

URL 中经常包含可变的部分，例如 /users/123 中的 123 代表用户 ID。Rocket 使用 <name> 语法来捕获这些动态部分，并自动转换为你指定的类型。

重要：Rocket 支持多种类型自动转换，包括：

  • 整数类型：u8, u32, i32, usize 等
  • 浮点类型：f32, f64
  • 字符串：String, &str
  • UUID：uuid::Uuid
  • 路径：PathBuf（捕获 /users/a/b/c 这样的多段路径）

查询参数

查询参数是 URL 中 ? 后面的部分，如 /search?q=rust&limit=10：

为什么用 Option？因为查询参数是可选的。Rocket 会：

  • 如果参数存在且类型匹配，提取值
  • 如果参数不存在，传入 None
  • 如果参数存在但类型不匹配（如 limit=abc），返回 400 错误

注意：? 是分界线，左边是路径参数，右边全是查询参数，两者永不交叉

路由优先级

当多个路由可能匹配同一个 URL 时，Rocket 采用"最具体优先"原则：

当访问 /users/all 时：

  • ① 完全匹配 → 选中
  • ② 也会匹配（因为 all 可以被当作 id），但因为①更具体，所以优先

当访问 /users/123 时：

  • ① 不匹配（路径是 /users/all 才对）
  • ② 匹配（123 是有效的 u32）→ 选中
  • ③ 也匹配，但②更具体

状态管理

HTTP 是无状态的——每个请求都是独立的。但很多应用需要跨请求共享数据：

  • 计数器（统计访问次数）
  • 数据库连接池
  • 配置信息
  • 缓存数据

Rocket 通过托管状态解决这个问题。

使用托管状态

关键理解：

  • manage() 注册状态，整个应用生命周期只有一个实例
  • 所有请求共享同一个状态（需要线程安全，用 Mutex、RwLock 或 Atomic）
  • State<T> 是只读引用，不能修改状态本身，但可以修改内部的可变数据

文件服务

文件服务在 Web 开发中很常见，但有个关键问题需要处理：浏览器对不同文件类型的行为不同。

  • 图片、PDF、Excel：浏览器会直接预览/打开，而不是下载
  • ZIP、EXE、RAR：浏览器会强制下载

但很多业务需求是无论什么文件，都让用户下载（比如安装包、数据导出）。这就需要针对不同场景采用不同的响应策略

基础文件服务

最简单的文件服务：直接返回 NamedFile。浏览器会根据文件类型决定行为

路径说明：文件路径基于当前工作目录：

  • 开发时：cargo run 执行的目录（通常是项目根目录）
  • 部署时：二进制文件所在的目录

NamedFile 本身不决定预览还是下载，它只是：

  1. 读取文件内容
  2. 自动设置正确的 Content-Type（根据文件扩展名）

真正决定浏览器行为的是 Content-Type 响应头：

Content-Type 浏览器行为
application/pdf 预览（打开 PDF 阅读器）
image/png 预览（显示图片）
video/mp4 预览（播放视频）
text/html 预览（渲染页面）
application/zip 下载（浏览器无法预览）
application/x-msdownload (.exe) 下载
application/octet-stream 下载（二进制流）

强制下载（二进制流方式）

设置 Content-Type: application/octet-stream（二进制流）

这种方式的原理是：告诉浏览器"这是一个二进制文件，我不知道它是什么类型"，浏览器就会强制下载。

特点：

  • ✅ 简单直接，无需自定义结构体
  • ✅ 强制下载，不会预览
  • ❌ 文件名丢失：下载后的文件通常叫 download（没有后缀名），用户需要手动改名

中间件（Fairing）

Fairing（整流罩）是 Rocket 的中间件系统。它让你可以在请求处理的不同阶段插入自定义逻辑，而不需要修改路由函数。

典型用途：

  • 日志记录（每个请求都打印日志）
  • 请求计时（统计 API 响应时间）
  • CORS 处理（添加跨域响应头）
  • 认证授权（检查登录状态）
  • 请求限流

Fairing 的生命周期

阶段 触发时机 用途示例
on_attach Rocket 启动时 验证配置、初始化资源
on_request 请求到达时 日志、认证、限流
on_response 响应返回前 添加响应头、压缩响应
on_shutdown 服务关闭时 清理资源、断开连接

异步编程

Rocket 的异步模型

Rocket 0.5 全面拥抱异步编程。这意味着：

  • 所有路由函数都可以是 async fn
  • 可以在处理函数中使用 .await 执行异步操作
  • Rocket 自动管理异步运行时（基于 tokio）

重要转变：

异步文件操作

同步操作会阻塞线程，异步操作则不会

CPU 密集型任务

异步擅长 I/O 操作（网络、文件、数据库），不擅长 CPU 密集型计算（图像处理、加密、排序）

概念 特性 静态链接库 动态链接库 链接时机 编译时 运行时 代码整合 库代码被直接复制到最终的可执行文件中。 库代码存在于独立的文件中，可执行文件只记录依赖关系。 文件后缀 Linux/Unix: .a (Archive) Windows: .lib Linux/Unix: .so (Shared Object) Windows: .dll (Dynamic Link Library) 可执行文件大小 较大，因为包含了所有依赖的库代码。 较小，因为不包含库代码本身。 内存占用 可能更高，因为每个使用该库的程序都在内存中有自己的副本。 可能更低，因为多个程序可以共享内存中的同一份库代码。 部署便利性 简单。可执行文件是独立的，不需要额外文件。 复杂。必须确保目标系统上有正确版本的库文件。 版本更新 困难。需要重新编译整个程序才能更新库。 容易。替换 DLL/.so 文件即可更新库，但需注意版本兼容性。 兼容性问题 无（因为库已绑定）。 可能存在“DLL Hell”，即版本冲突或缺失。 在rust中是通过toml中的设置来选择库类型 ...
