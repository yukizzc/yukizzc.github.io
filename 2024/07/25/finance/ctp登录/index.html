接口

Spi （如 CThostFtdcTraderSpi），包含有所有的响应和回报函数，用于接收综合交易平台发送或交易所发送 综合交易平台转发的信息。开发者需要继承该接口类，并实现其中相应的虚函数

Api （如 CThostFtdcTraderApi），包含主动发起请求和订阅的接口函数，开发者直接调用即可

初始化

  1. 创建实例：创建 SPI（CThostFtdcTraderSpi)和 API（CThostFtdcTraderApi）实例

  2. 注册 SPI：向 API 实例注册 SPI 实例

  3. 注册前置地址：交易接口注册交易前置，行情接口注册行情前置

  4. 订阅公有流（仅交易接口）：接收公有数据（合约状态等），模式：Resume（默认）/Restart/Quick

  5. 订阅私有流（仅交易接口）：接收私有数据（报单回报等），模式：Resume（默认）/Restart/Quick

  6. 初始化：调用 Init()

  7. 等待线程退出：调用 Join()

登录、登出

整个ctp登录过程如下

当 API 实例调用 Init() 方法时，系统会尝试连接柜台，连接成功后，前置回调函数 OnFrontConnected() 将会被触发，然后去执行登陆结算等

主动发送消息的流程：创建数据对象 → 设置属性 → 调用 API 主动函数发送。后面所有ctp操作其实都是这样的过程

属性说明

  • GetTradingDay：这个是api中自带的方法，通过它可以返回当前交易日，对于夜盘晚上会返回下个交易日，如当天是否确认结算单等就可以靠这个来标记。登录回调的TradingDay也是一样的作用

  • FrontID, SessionID：这两个属性在OnRspUserLogin回报中返回，分别是前置编号和会话编号，其中后者每次重新登录都会不同并且是递增的。许多人撤单会使用 FrontID + SessionID + OrderRef 撤单的方式，不过我个人喜好第二种方式不使用这两个ID而是采用柜台实际编号的方式，具体后面下单环节再细说

流控

CTP交易系统基于安全和性能考虑，在诸多地方有流量控制，其中流量控制又分FTD报文流量控制、报单流量控制、查询流量控制等。而这些流量控制分布在各个不同的地方

报单流控

报单流控是指用户在本交易系统报单（ReqOrderInsert）、撤单（ReqOrderAction)时每秒内允许的最大笔数。报单流控限制配置在CTP柜台端【程序化交易频繁报撤单管理】菜单。如果超过这个限制API会通过OnRspOrderAction提示：“CTP:下单频率限制”

查询流控

自从穿透式监管版本以后，API在连接交易前置时，会去查询到前置的查询流控设置（该设置配置在front_se组件）。假设前置配置了2笔/秒，那么连接该前置的API每秒只能发起2笔查询请求。但是要注意，不管怎么配置，API都内置了在途流控，在途查询流控为1笔。即，当前这笔查询请求发出后，在未收完所有的查询响应前，不能发起下一笔查询请求。在过去，查询流控是内置在API里，1笔每秒，在途1笔。如果超过交易前置配置的查询流控，则会触发OnRspError，并提示：“CTP：查询未就绪，请稍后重试”如果超过API内置的在途流控，则查询请求的返回值为-2，表示未处理请求超过许可数

以上是官方文档中的说明，本地使用下是感觉同时几十个报单报出去没有收到流控，但是查询不能无脑查。我目前设计分两种方式去查询

低频次查询

适用于保证金率查询、合约信息查询等低频操作。采用内部循环同步重试方式：

  • 根据查询函数的返回值判断
  • 返回 0 表示成功，其他值表示失败
  • 失败时延迟 1 秒后重新发送请求

高频次查询

在高频交易场景中，查询操作（如委托回报查询）如果使用同步方式，会在大量委托时阻塞主线程，影响后续消息的接收和处理。为了解决这个问题，设计了一个独立的异步任务处理器，将查询操作放入单独的线程中执行，使用独立的事件循环，实现异步非阻塞的查询

查询函数

查持仓

ReqQryInvestorPosition 用于查询投资者持仓，查询结果通过回调函数 OnRspQryInvestorPosition 逐条返回。

查询参数：

  • 设置品种合约：只返回该品种的持仓
  • 不设置品种合约：返回所有持仓合约

关键字段

字段 说明
PositionDate 持仓日期：‘1’ = 当前交易日持仓（今仓），‘2’ = 历史持仓（老仓） 注：仅上期所和能源中心有此字段
PosiDirection 持仓方向：‘2’ = 多头，‘3’ = 空头

特殊说明

上期所的今仓和老仓会作为两条独立记录返回，其他交易所的今老仓合并为一条记录。

持仓唯一标识设计

为区分不同持仓，个人使用组合键作为唯一标识：

示例： 12:rb2410 表示螺纹钢10合约的多头今仓

持仓信息字典如下

  • 对于上海品种，老仓记录下只有 Position（总持仓），TodayPosition（今持仓）为 0；而今仓记录下 Position 和 TodayPosition 相等。对于非上海品种，Position 就是总持仓，TodayPosition 表示今仓数量。因此平仓时直接读取 Position 即可，一般无需区分今老仓。
  • 关于冻结字段，LongFrozen（多头冻结）和 ShortFrozen（空头冻结）表示挂单占用的数量。对于多头持仓，LongFrozen 是开多未成交，ShortFrozen 是平多未成交；对于空头持仓，ShortFrozen 是开空未成交，LongFrozen 是平空未成交。
  • 因此计算实际可平数量时，需要根据持仓方向扣除对应的平仓挂单冻结：多头持仓的可平数量为 Position - ShortFrozen（总持仓减去平多挂单），空头持仓的可平数量为 Position - LongFrozen（总持仓减去平空挂单）。

查报单

  • 通过 ReqQryOrder 查询报单，该函数支持传入市场、品种、OrderSysID 编号、时间等作为过滤条件。执行后回调函数 OnRspQryOrder 返回查询结果。
  • 如果需要定向查询特定类型的报单（如开多单），可以将开平动作和买卖方向组合后作为 nRequestID 传入，然后在回调函数中根据 nRequestID 来区分本次查询的是哪类报单。
