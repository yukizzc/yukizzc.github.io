HTTP协议

基础概念

HTTP 协议是基于 TCP/IP 协议栈 的应用层协议，其有如下特点

  • 无状态协议：每个请求独立，服务器不保留客户端状态
  • 基于请求/响应模型：客户端发送请求，服务器返回响应
  • 可扩展：通过头字段支持各种功能扩展
  • 支持多种数据格式：通过 Content-Type 指定

Socket 的作用

Socket 的作用可以理解为**“通信的抽象接口”**，它在网络协议栈和应用层之间架起了一座桥梁，让开发者能够更简单地实现网络通信

  • Socket 位于应用层（用户进程）和传输层（TCP/UDP）之间，是一个抽象层
  • 它屏蔽了底层协议（如 TCP/IP、ARP、物理介质等）的复杂性，开发者无需关心数据如何拆分、封装、传输，只需调用 Socket 提供的接口即可

URL 结构

一个完整的URL包括：协议、ip、端口、路径、参数

例如：https://www.baidu.com/s?wd=sanqiansi

  • https 是协议
  • www.baidu.com 是域名（DNS解析为IP）
  • 端口默认 443（HTTPS）或 80（HTTP）
  • /s 是路径
  • 参数是 wd=sanqiansi

GET 与 POST 的区别

  • GET：提交的数据会放在URL之后，以?分割URL和传输数据，参数之间以&相连，如EditBook?name=test1&id=123456
    • 数据大小有限制（因为浏览器对URL的长度有限制）
  • POST：提交的数据放在HTTP包的请求体中
    • 数据大小没有限制

HTTP 报文格式

HTTP 请求格式

一个完整的HTTP请求由三部分组成：请求行 (Request Line)、请求头 (Request Headers) 和请求体 (Request Body)。它们之间通过换行符（通常是 \r\n）来分隔

请求行 (Request Line)

第一行，包含三个基本元素，用空格分隔：

  • 方法 (Method)：操作类型，如 GET, POST, PUT, DELETE, HEAD 等
  • 请求URI (Request-URI)：要访问的资源路径，例如 /index.html 或 /api/users
  • HTTP版本 (HTTP Version)：使用的HTTP协议版本，如 HTTP/1.1 或 HTTP/2

请求头 (Request Headers)

从第二行开始，到第一个空行之前，都是请求头。请求头是键值对（Key-Value）的形式，用来向服务器传递关于请求和客户端本身的附加信息。

  • 格式：Header-Name: Header-Value
  • 每个头字段占一行

常见请求头：

请求头 说明
Host: www.example.com HTTP/1.1 必需，指定服务器的主机名和端口号
User-Agent: Mozilla/5.0... 标识客户端软件（浏览器、爬虫等）
Content-Type: application/json 指明请求体的媒体类型
Content-Length: 123 指明请求体的字节大小
Authorization: Bearer <token> 身份验证的凭据
Accept: application/json 告诉服务器客户端希望接收的响应类型
Cookie: name=value 发送之前服务器设置在本地的Cookie

请求体 (Request Body)

在请求头之后的一个空行（即连续的两个 \r\n）之后，就是请求体。

  • 作用：携带需要发送给服务器的数据，例如表单提交的内容、JSON数据、文件二进制流等
  • 注意：GET, HEAD, DELETE 等方法通常没有请求体；POST 和 PUT 等方法通常有

HTTP 响应格式

当服务器处理完请求后，会返回一个HTTP响应。结构类似，由三部分组成：状态行 (Status Line)、响应头 (Response Headers) 和响应体 (Response Body)

Content-Type 详解

Content-Type 是 HTTP 协议中用于标识请求或响应中传输的数据类型的头部字段

表单数据格式

Content-Type 用途 示例
application/x-www-form-urlencoded 默认的表单提交格式，数据被编码为键值对 name=John&age=30
multipart/form-data 用于上传文件或二进制数据，数据分成多个部分传输 包含边界分隔符和二进制数据

结构化数据格式

Content-Type 用途 示例
application/json 传输JSON格式数据，广泛用于RESTful API {"name": "John", "age": 30}
application/xml 或 text/xml 传输XML格式数据，常见于传统企业级应用 <user><name>John</name></user>

纯文本或二进制格式

Content-Type 用途 示例
text/plain 传输纯文本数据 This is a plain text message.
application/octet-stream 传输任意二进制数据 文件二进制流

其他专用格式

  • application/pdf、image/jpeg、video/mp4 等，直接传输特定类型的文件

前后端开发模式

传统模式：前后端不分离（服务端渲染）

流程如下：

  1. 用户请求：用户在浏览器输入网址，发起一个 HTTP 请求。
  2. 服务端处理：后端服务器（如 Java Spring, PHP, Python Django, .NET）接收到请求。
  3. 执行业务逻辑：后端服务会执行相应的业务逻辑，其中包括从数据库查询数据。
  4. 渲染HTML：后端使用模板引擎（如 JSP, Thymeleaf, Freemarker, Blade, Django Template）将查询到的数据嵌入到 HTML 模板中，生成一个完整的、包含了数据和样式的 HTML 字符串。
  5. 返回响应：服务器将这个完整的 HTML 文档作为响应返回给浏览器。
  6. 浏览器展示：浏览器接收到 HTML 后，直接解析并展示给用户。

现代模式：前后端分离（客户端渲染）

这种模式下后端只负责提供数据接口（API），前端负责展示和交互，两者通过 HTTP 请求（通常是 JSON 格式）进行通信。流程如下：

  1. 用户请求：用户浏览器输入网址，请求一个纯粹的 HTML文件（这个文件内容很简单，主要包含了一些用于构建页面的 JavaScript 和 CSS 链接）。
  2. 返回静态资源：服务器（可能是 Nginx、CDN 等静态文件服务器）返回这个"空壳"HTML文件。
  3. 执行JS，调用API：浏览器解析HTML，加载其中的 JavaScript 框架（如 React, Vue, Angular）代码。这些 JS 代码开始执行，并向后端服务器发起 AJAX/Fetch 请求（例如 /api/users）。
  4. 后端处理并返回数据：后端服务器（如 Java SpringBoot, Node.js Express）接收到 /api/users 请求，同样执行业务逻辑、查询数据库，但不再渲染HTML，而是将数据以 JSON 格式返回给前端。
  5. 前端渲染：前端 JavaScript 接收到 JSON 数据后，动态地操作 DOM（使用 React/Vue 的虚拟DOM技术），将数据渲染到页面中相应的位置。

两种模式对比

特性 前后端不分离（服务端渲染） 前后端分离（客户端渲染）
核心职责 后端负责数据+页面渲染 后端只提供数据，前端负责渲染
通信方式 服务器返回完整HTML 前端通过API调用获取JSON数据
耦合度 高，前后端代码混杂 低，通过API接口解耦
开发模式 前后端协同开发，效率可能较低 前后端并行开发，效率高
用户体验 整页刷新，体验一般 局部刷新，SPA体验流畅
SEO 非常友好 不友好，需要额外技术（SSR）优化
典型技术 JSP, Thymeleaf, Django Template React, Vue, Angular + RESTful API
适用场景 官网、博客、内容型网站等注重SEO的场景 后台管理系统、WebApp、交互复杂的场景

RESTful API 设计风格

RESTful 是一种 API 的设计风格和规范，而不是一个协议或标准。它的核心思想是用HTTP协议的特性来直观地表达对服务器资源的操作

HTTP 方法 操作类型 描述 举例（含义）
GET Read（读取） 安全地获取资源（一个或多个） GET /users (获取所有用户) GET /users/123 (获取ID为123的用户)
POST Create（创建） 在服务器创建一个新资源 POST /users (创建一个新用户)
PUT Update（更新） 完整更新一个已存在的资源（提供资源的全部信息） PUT /users/123 (完整更新ID为123的用户信息)
PATCH Update（更新） 部分更新一个已存在的资源（只提供需要修改的信息） PATCH /users/123 (只更新用户的邮箱)
DELETE Delete（删除） 删除一个指定的资源 DELETE /users/123 (删除ID为123的用户)

注意：这只是约定俗成的规范，通过不同的方法来表示执行对应的动作。虽然没有硬性规定，但这样设计更友好和通用。

Pydantic 数据验证

Pydantic 是一个 Python 数据验证和设置管理的库，核心功能如下：

  1. 数据验证：确保输入数据符合预期格式
  2. 类型提示：利用 Python 类型提示
  3. 设置管理：管理配置和设置
  4. 序列化/反序列化：JSON ↔ Python 对象转换

定义数据模型类似 Rust 中的结构体，用来定义字段及其数据类型

FastAPI 参数传递详解

路径参数

在 FastAPI 中，使用花括号 {} 定义的路径片段是动态参数，作为变量传递给路径操作函数，此时路由函数里必须有对应的参数。

客户端通过 http://127.0.0.1:8000/user/5 传递具体参数。路由函数中的类型声明可以起到自动转换作用，如果是不可转换类型，客户端访问时会提示错误。

注意：HTTP 传输的数据都是字符串格式，FastAPI 通过类型注解自动进行数据验证和转换：字符串 → 验证 → 目标类型

查询参数

查询参数就是 URL 中 ? 后面的键值对。比如百度电影地址：https://top.baidu.com/board?platform=pc&tab=movie

查询参数不需要在路由路径中定义，只需要在路径操作函数的参数中声明即可。

  • 可以指定默认值（客户端可以不传）
  • 默认值用 None 表示可选（客户端不传参时返回 null）
  • 无默认值的参数必须在访问时提供

访问示例：http://127.0.0.1:7777/user/?user_id=5&name=zzc

请求体（JSON）

路径参数和查询参数都写在 URL 中，一方面不安全，另一方面也受限于输入长度限制，此时可以把数据放到请求体中。

规则：

  • 如果参数是 Pydantic 模型类型，FastAPI 会强制将其视为请求体参数
  • 如果是常规简单类型，FastAPI 会尝试从查询参数/路径参数中寻找

WebSocket 实时通信

概述

  • 全双工通信协议，允许客户端和服务器之间进行实时双向通信
  • 相比 HTTP 的请求-响应模式，WebSocket 提供持久连接
  • 适用于实时应用：聊天、通知、实时数据更新等

WebSocket 服务端工作流程：

  1. 客户端发起 WebSocket 连接请求
  2. 服务端接受连接 (accept)
  3. 进入通信循环：
    a. 接收客户端消息 (receive)
    b. 处理业务逻辑
    c. 发送响应给客户端 (send)
  4. 连接断开，清理资源

HTTP 与 WebSocket 对比

特性 HTTP WebSocket
连接方式 短连接，每次请求新建 长连接，一次建立持续使用
通信方向 客户端主动发起 双向实时通信
延迟 较高（每次握手） 极低（握手一次）
适用场景 网页浏览、API调用 实时聊天、股票行情、在线游戏
