python并发异步

Created2024-07-26|Updated2026-03-28|computer
|Post Views:

概要

  • 多线程：threading，利用CPU和I/O可以同时执行的原理，让CPU不会干巴巴等待I/O完成

  • 多进程：multiprocessing，利用多核CPU的能力，真正的并行执行任务

  • 异步IO：asyncio，在单线程利用CPU和I/O同时执行的原理，实现函数异步执行

  • 使用Lock对资源加锁，防止冲突访问

  • 使用Queue实现不同线程/进程之间的数据通信，实现生产者-消费者模式

  • 使用线程池Pool/进程池Pool，简化线程/进程的任务提交、等待结束、获取结果

  • 使用subprocess启动外部程序的进程，并进行输入输出交互

CPU和I/O密集型区别

  • CPU密集型（CPU-bound）：

CPU密集型也叫计算密集型，是指I/O在很短的时间就可以完成，CPU需要大量的计算和处理，特点是CPU占用率相当高

例如：压缩解压缩、加密解密、正则表达式搜索

  • IO密集型（I/O bound）：

IO密集型指的是系统运作大部分的状况是CPU在等I/O (硬盘/内存) 的读/写操作，CPU占用率仍然较低。

例如：文件处理程序、网络爬虫程序、读写数据库程序

进程、线程、协程

特性 多线程 (threading) 多进程 (multiprocessing) 多协程 (asyncio)
实现方式 多个线程在同一进程内运行 多个独立进程运行 单线程内通过事件循环调度协程
资源共享 共享内存，需锁机制（如 Lock）避免竞争 进程间内存独立，需 Queue 或 Pipe 通信 共享内存，无需锁，依赖事件循环管理
并发/并行 并发（受 GIL 限制，非真正并行） 并行（真正利用多核 CPU） 并发（单线程内切换执行）
适用场景 I/O 密集型任务（如网络请求、文件操作） CPU 密集型任务（如计算密集型操作） I/O 密集型任务（高并发网络应用）
开销 较小（线程切换快） 较大（进程创建和通信成本高） 极小（协程切换由用户态控制）
GIL 影响 受 GIL 限制，多核利用率低 不受 GIL 影响，可充分利用多核 不受 GIL 影响（单线程运行）
异常处理 线程独立，需手动捕获 进程独立，需通信机制传递异常 协程异常可集中处理（如 gather）
编程复杂度 中等（需处理锁和同步） 较高（需管理进程间通信） 中等（需理解 async/await 语法）
启动方式 Thread(target=func) Process(target=func) asyncio.run(main())
典型库/工具 threading, concurrent.futures multiprocessing, concurrent.futures asyncio, aiohttp

总结

  • 多线程适合轻量级 I/O 任务，但受 GIL 限制
  • 多进程适合需要并行的 CPU 任务，开销大但无 GIL 限制
  • 多协程适合高并发 I/O 任务，开销最小但需异步编程思维

锁

基本概念：锁是一种同步原语，用于确保在同一时间只有一个线程可以访问共享资源

工作机制：当一个线程获得锁后，其他试图获取锁的线程将被阻塞，直到该锁被释放

适用场景：适用于需要保护的共享资源较多，或者需要确保一段代码块在某一时刻只有一个线程执行时

事件

threading.Event 是 Python threading 模块提供的一种线程同步原语（synchronization primitive）。它用于线程之间的通信，允许一个或多个线程等待某个事件的发生，而另一个线程负责触发这个事件。简单来说，它是一个“信号机制”：

  • 初始状态：Event 对象创建时，默认是“未触发”（未设置，False）
  • 等待：线程可以调用 event.wait()，如果事件未触发，线程会阻塞（暂停执行），直到事件被触发
  • 触发：某个线程调用 event.set()，将事件设置为“已触发”（True），此时所有等待该事件的线程会被唤醒，继续执行

核心方法

  • event.wait()：阻塞当前线程，直到事件被触发（即 event.set() 被调用）。如果事件已经触发，则立即返回
  • event.set()：将事件设置为“已触发”状态，唤醒所有等待该事件的线程
  • event.clear()：将事件重置为“未触发”状态（代码中未使用，但很重要）
  • event.is_set()：检查事件当前是否已触发，返回布尔值

Condition

用途：更复杂的同步，线程等待特定条件并在条件满足时被唤醒

核心方法：

  • acquire() / release()：获取/释放锁
  • wait()：等待条件
  • notify() / notify_all()：通知一个/所有等待线程

异步

概念

  1. 异步是为了在等 I/O（网络、文件、数据库）时不傻等，可以去做别的事。
  2. 异步代码长得像同步代码，但多了 async / await 两个关键字。
  3. 所有异步程序必须用 asyncio.run(…) 启动。

关键字：

关键字 作用 位置
async def 定义可以暂停的函数（协程函数） 函数定义前
await 暂停当前函数，去等另一个异步操作 只能在 async def 里面
asyncio.run() 程序的真正入口，启动整个异步世界 最外层，main 函数调用
asyncio.create_task() 把协程立刻扔到后台跑（真正并发） async def 里面常用
  • asyncio.run()
    • 不开启新线程，完全在当前线程执行，每次执行会创建全新的事件循环运行时
    • 只接收一个协程对象作参数，但可以把多个协程运行塞到另一个协程函数里然后被调用（参考下面运行模板）
    • 结束时自动清理任务和队列
    • 不需要开线程所以可以频繁创建，消耗极小
  • create_task()
    • 将一个协程包装成任务（Task）
    • 立即放入事件循环的任务队列，开始执行
    • 返回一个 Task 对象，可以稍后获取结果
    • 实现真正的并发执行
  • asyncio.gather()
    • 并发执行多个协程
    • 等待所有完成（或第一个出错）
    • 按传入顺序返回结果
    • 会自动将协程转为任务

运行模板

同步函数

  • 异步函数（async def）本身不会阻塞事件循环，但必须运行在 asyncio 事件循环中。
  • 同步函数（def）所在的线程默认没有运行中的事件循环，因此需要“桥接”机制来执行异步代码。
  • 所有桥接方式的最终目标：让同步代码能安全、方便地获取异步函数的执行结果。
  • 是否阻塞调用线程，取决于具体方式，而不是必然的。
方面 方式1：后台线程 + 永久事件循环 方式2：asyncio.run() 临时事件循环
是否新开线程 是（新开一个守护线程专门运行 loop.run_forever()） 否（直接在当前线程运行）
事件循环生命周期 长寿命：创建后一直存在，直到程序结束 临时：用完即关（创建 → 运行 → 关闭）
是否阻塞当前线程 可以选择：
• 立即 .result() → 阻塞
• 先提交多个再等 → 可并发		 总是阻塞当前线程，直到协程完成
是否支持并发多个任务 支持（先 submit 多个任务，再统一等待，实现并发） 不支持（一次只能运行一个顶层协程，多个要嵌套 await）
线程安全 高（可以从任何线程安全提交任务） 低（只能在没有运行中事件循环的线程中使用）
使用复杂度 稍高（需要自己管理类实例） 极低（一行 asyncio.run() 搞定）
适用场景 • 同步代码中频繁调用异步函数
• 需要从多个线程调用
• 需要并发执行多个异步任务
• 程序长期运行（如 Web 服务、后台脚本）		 • 只需要偶尔调用一次异步函数
• 程序入口简单（如脚本、命令行工具）
• 不需要并发多个独立异步任务
是否“临时” 不是（事件循环是永久的） 是（真正的一次性临时事件循环）

临时事件

  • 会阻塞当前线程，直到异步任务完成。
  • 用完即关，最干净、最安全（Python 官方推荐，3.7+）。
  • 不支持并发多个独立异步任务（如果想并发，必须在 async_task 内部嵌套 await 多个子任务）。
  • 不能在已经运行事件循环的线程中使用（如异步函数内部不能再调用 asyncio.run()）。

后台常驻

  • 新开一个守护线程，事件循环永久运行。
  • submit() 是线程安全的，可从任何线程调用。
  • 如果立即调用 .result() → 阻塞（表现像同步调用）。
  • 如果先提交多个任务，再统一等待 → 可实现真正并发（总时间取最长任务）。
  • 适合在 Flask/Django 同步视图、线程池、长运行脚本中使用。

多任务并行

  1. 把多个异步函数包装成 asyncio.Task（通常用 asyncio.create_task()）。
  2. 在一个异步函数中收集这些 task。
  3. 用 await asyncio.gather(*tasks) 或逐个 await task 来等待它们完成。
  4. 在同步代码中，通过 asyncio.run(main_async()) 来启动这个主异步函数。

把异步包装成task然后用后台常驻模式就可以启动并行并且不阻塞当前线程

异步中的同步阻塞

如果在异步函数中执行同步阻塞，虽然可以不阻塞主线程，但是会阻塞整个事件循环。这也是异步的关键点，整个任务队列中但凡有一个同步阻塞，整个事件循环内都会被堵塞，导致结果是无法做到并行。

解决方法就是获取当前运行时，使用loop.run_in_executor把同步函数扔到线程池中。3.9后的新版直接用asyncio.to_thread完成该工作

总结

  1. 全异步链路串行执行：await 确实是串行的，等待当前协程完成才继续
  2. 全异步链路并行执行：创建多个 Task 并使用 gather/wait 实现并行
  3. 同步函数调用异步：
    • asyncio.run()：临时创建运行时（适用于脚本、简单调用）
    • 后台线程 + run_forever()：不阻塞主线程
  4. 异步函数中调用同步阻塞函数：
    • 直接调用会阻塞事件循环
    • to_thread() 将阻塞操作移到线程池执行

协程和任务的一个区别

操作 是否阻塞当前协程 说明
await coroutine() 是 直接等待协程完成，当前协程暂停
task = asyncio.create_task(coroutine()) 否 立即返回 task，当前协程继续执行，协程在后台启动
await task 是 等待这个 task 完成，当前协程暂停（即使之前用了 create_task）
create_task(...) 后不 await 否 真正的“后台并发”
